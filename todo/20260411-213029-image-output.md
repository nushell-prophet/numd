# Spec: image output option

Add a fence option that renders a code block's captured output as a PNG
image instead of embedding it as plain text (`# =>` inline lines or an
`output-numd` fenced block). The image is referenced from the markdown via
a standard `![](path)` link.

## Motivation

Plain-text captures lose ANSI colors, table borders rendered via unicode
box-drawing, and any styling `table -e` applies. Screenshots of the
rendered terminal are how people currently share nushell output on blogs
and social media. An option that does this automatically from inside numd
removes the manual screenshot step and keeps the artifact in sync with
the code.

## The `to png` plugin

`to png` is provided by a nushell plugin (confirmed interactively in the
sandbox; need to document the exact plugin source in README install
instructions before shipping). Contract:

- input: `string` — must already contain ANSI escapes, i.e. the caller
  has to pipe through `table -e` (or any command that emits colored
  output) before `to png`
- positional `output-path: path` (optional) — file path to write. If
  omitted, the plugin picks a timestamped filename in cwd
- output: `string` — the absolute path of the written PNG
- relevant flags: `--width <int>`, `--theme <string>`, `--font <string>`,
  plus custom color/font overrides

Implication: the intermediate script must execute each image-producing
group as `<user pipeline> | table -e --width W | to png <path> | ignore`,
and the **path** is decided by numd before the script runs so the
resulting markdown can reference it deterministically.

## Fence option

- long form: `image`
- short form: `i`
- added to the `fence_options` const in `numd/commands.nu:417`
- added to `convert-short-options` in `numd/commands.nu:437`

Example:

    ```nu image
    ls | first 5
    ```

After `numd run`, the block becomes:

    ```nu image
    ls | first 5
    ```

    ![](media/<doc>.block-3-0.png)

## Output file layout

- Directory: a single `media/` folder sibling to the markdown file.
  All numd docs in the same folder share it.
  **Why one flat folder instead of one per doc:** simpler — no compound
  directory name per document, matches the `media/` / `assets/` / `static/`
  convention common in static-site generators, and the per-doc prefix in
  the filename still prevents collisions.
  Override via `$env.numd.image-dir` (path, absolute or relative to the
  markdown file). Default: `media`.
- Filename: `<doc-stem>.block-<block_index>-<group_index>.png`.
  Example: `README.md` → `media/README.block-3-0.png`.
  `doc-stem` is the markdown filename without extension and carries the
  disambiguation that used to live in the directory name.
  `block_index` is the 1-based index already assigned by
  `parse-markdown-to-blocks`; `group_index` is the position of the
  command group inside the block (blocks are split by blank lines in
  `process-code-block-content`, so a block can have multiple groups and
  thus multiple images).
  **Why deterministic names:** keeps git diffs small — re-running numd on an
  unchanged block overwrites the same PNG instead of creating a new file.
  **Not content-hash-based because:** a trivial edit would leave the old PNG
  orphaned on disk, and image references in the markdown would churn on
  every edit.

## Rendering pipeline changes

In `numd/commands.nu`:

1. **`format-command-output` (line 220)** — add a branch:

       if 'image' in $fence_options { generate-image-output-pipeline <path> }

   placed **before** the `separate-block` branch. `image` wins over
   `separate-block` when both are set (see Conflicts below).

2. **New `generate-image-output-pipeline`** — mirrors
   `generate-inline-output-pipeline` (line 589) but produces:

       do {
           $env.config.use_ansi_coloring = true
           $in
           | table -e --width ($env.numd?.table-width? | default 120)
           | to png ($path)
           | ignore
       }

   Note `table -e` (expand), not plain `table`, so nested structures
   render fully before being rasterized. `ignore` drops the returned
   path string so nothing lands in captured stdout for this group.

   **Why the `do { ... }` wrapper forcing `use_ansi_coloring = true`:**
   `to png` requires its input to already contain ANSI escapes, but
   the intermediate script runs under `nu -n` where
   `$env.config.use_ansi_coloring` defaults to `false` in
   non-interactive mode, so `table -e` would emit plain text and `to
   png` would rasterize an uncolored block. Mutating inside `do` keeps
   the override scoped to this pipeline only (verified empirically
   that the `# =>` inline and `separate-block` paths stay ANSI-free,
   so no defensive `ansi strip` is needed elsewhere). Mutation rather
   than replacement preserves the rest of `$env.config`.
   `with-env {FORCE_COLOR: '1'}` does not work here because nu reads
   that env var at process startup, not per-pipeline.

3. **Path resolution** — `decorate-original-code-blocks` (line 251)
   already has `block_index`. It needs to also know the group index per
   group. The easiest seam is inside `process-code-block-content` (line
   282): when `image` is in `fence_options`, pass an enumerated index
   into each `generate-block-execution` call, and thread it down to
   `format-command-output` so the generated pipeline knows its target
   path. `group_index` is 0-based over *all* groups in the block, not
   just executable ones, to keep numbering stable if a user later adds a
   comment-only group in the middle.

4. **Image reference emission** — `![](...)` must land in the text
   *after* the closing code fence of the block, not between groups.
   A multi-group block emits a single closing `` ``` `` once, so a
   literal "`print` the reference after each group" would interleave
   `![](...)` lines *inside* the code fence and produce invalid
   markdown. Collect the references at the block level in
   `decorate-original-code-blocks` (line 251) or within the generator
   used by `process-code-block-content`, then hand them to
   `generate-block-markers` (line 636) which already emits the
   closing fence — append each `![](rel_path)` line on new lines
   right after the closing `` ``` `` (and before the block's
   terminating blank line) so they render in the captured stream
   between `#code-block-marker-open-N` and
   `#code-block-marker-close-N`. That way `extract-block-index` +
   `merge-markdown` route them to the right spot in the final
   document.

   `$rel_path` is the image path relative to the markdown file (not
   absolute), so the document stays portable when committed or
   moved.

   **Re-run duplication guard.** Because the references live in
   *text* blocks following the code block (not inside the code
   block itself), re-running numd would leave the old `![](...)`
   lines in place and append new ones on every invocation. To
   prevent accumulation, `parse-file` must strip any line matching
   `![](<image-dir>/<stem>.block-\d+-\d+\.png)` *before* classifying
   blocks. Locate the stripper in `parse-file` (not inside
   `execute-blocks`) so it runs on both the `run` path (idempotent
   re-runs) and the `clear-outputs` path (user-facing clear). Do
   **not** delete the PNG files on disk — the Clear-outputs section
   below covers that.

5. **Directory creation** — `execute-blocks` (or its caller in
   `numd/mod.nu`) ensures `media/` (or the `$env.numd.image-dir`
   override) exists as a sibling of the markdown file before running
   the intermediate script. Skipped entirely if no block in the doc
   uses the `image` option, so running numd over a doc with no image
   blocks never creates the folder.

6. **Plugin loading in the child nu process** —
   `execute-intermediate-script` (line 473) currently runs the child
   as `^$nu.current-exe -n <script>` under the default (non
   `--use-host-config`) path. `-n` turns off the config and env
   files but also prevents the plugin registry from being loaded,
   so `to png` is not available in the child even when
   `--plugin-config $nu.plugin-path` is passed.

   Verified empirically: `nu -n --plugin-config <registry>` does not
   expose `to png`, but `nu -n --plugins=<exe-path>` does, and the
   full `'hello' | table -e | to png /tmp/x.png` pipeline succeeds
   under that second form.

   **Change:** when any block in the doc uses `image`, have
   `execute-intermediate-script` additionally pass
   `--plugins=<exe-path>` for the `to png` plugin executable. The
   path is discovered in the parent numd process at run-time via:

       plugin list
       | where {|p| 'to png' in ($p.commands.name | default []) }
       | get filename.0

   If that query returns nothing, the plugin precheck (see Plugin
   availability) fails with the install message instead of letting
   the child script crash. The parent runs in whatever nu the user
   invoked numd under, so its plugin registry *is* loaded and the
   discovery is reliable.

   **Use the `--plugins=<path>` equals form, not
   `--plugins <path>` as two separate args.** `--plugins` is
   declared as variadic (`path...`), which means when there is a
   trailing script-file positional it greedily absorbs the script
   path as a second plugin argument and errors with
   "valid Nushell plugin filenames must start with nu_plugin_".
   The equals form binds exactly one value to the flag and leaves
   the script-file positional alone. Example that works:

       ^$nu.current-exe -n $'--plugins=($exe_path)' $intermed_script_path

   **Why `--plugins` and not `--plugin-config`:** `-n` suppresses
   the plugin registry file regardless of `--plugin-config`, so
   `--plugins` (explicit executables) is the only way to keep the
   existing reproducibility guarantees of `-n` mode while still
   making `to png` available. Only the plugin we need is injected;
   the user's full plugin set is not loaded.

   **Why not require `--use-host-config`:** that flag loads env,
   config, and the plugin registry together, which defeats the
   reproducibility goal that motivated the `-n` default in the
   first place. Threading a single plugin executable through is a
   smaller deviation.

## Interaction with other fence options

| combined with     | behavior                                                        |
|-------------------|-----------------------------------------------------------------|
| `no-run` / `N`    | No execution, no image. Existing image files left untouched.    |
| `no-output` / `O` | No image. `no-output` wins — matches its current early return.  |
| `separate-block`  | `image` wins; the output-numd fence is not emitted.             |
| `try` / `t`       | Error value is rendered by `table -e` then rasterized normally. |
| `new-instance`    | Unchanged; the image pipeline runs in the spawned instance.     |
| `run-once`        | First run produces the image; block is then flipped to no-run.  |

## Config knobs

Read from `$env.numd` with fallbacks:

- `$env.numd.image-dir` — override output directory
- `$env.numd.image-theme` — `to png --theme` value (default: plugin default)
- `$env.numd.image-font` — `to png --font` value
- `$env.numd.table-width` — already exists; reused for `--width`

Keep the knob list short on the first pass. Custom per-color overrides
can come later if anyone asks.

## Plugin availability

`to png` is an external plugin, not a builtin. Fail fast: before
executing, scan fence options, and if any block uses `image`, verify
`to png` is registered by querying `plugin list` in the parent numd
process:

    plugin list
    | where {|p| 'to png' in ($p.commands.name | default []) }
    | get filename.0?

If the result is empty, error with a clear install message naming the
plugin. If it returns a path, stash it — it becomes the `--plugins`
argument passed to the child nu process (see Rendering pipeline step
6). Do the check and discovery once per `numd run` invocation, not per
block.

Use `plugin list` rather than `scope commands` because `scope commands`
only tells us whether the command is registered, not where the
executable lives — and we need the executable path in order to inject
the plugin into the `-n` child.

## Clear-outputs behavior

`numd clear-outputs` strips `# =>` lines and `output-numd` fences,
and the same run needs to strip trailing `![](.../block-<i>-<g>.png)`
image reference lines as well. Do **not** delete the PNG files from
disk — image files are user-visible artifacts and deletion from a
"clear outputs" command would be surprising. A separate `numd
clean-images` helper can come later if needed.

**Single implementation seam:** the image-reference stripper lives in
`parse-file` (see Rendering pipeline step 4, "Re-run duplication
guard"), not in a separate `strip-outputs`-style function. That way a
single regex-based filter serves both the `run` path (idempotent
re-runs — the parser drops old refs before classifying blocks, so the
pipeline can regenerate fresh ones without duplication) and the
`clear-outputs` path (user-facing clear — the same parser pass runs
before `strip-outputs`, so image references are gone by the time the
markdown is reassembled). Splitting this across two functions would
duplicate the regex and drift.

`clear-outputs --strip-markdown` drops the image reference line along
with all other markdown content, so the resulting `.nu` script contains
just the code. No special handling is needed for image blocks in that
path — the existing markdown-strip behavior is already correct.

## Non-goals

- SVG or other formats — PNG only for now.
- Diffing old vs new images — git already handles binary diffs.
- Rendering the *code* as an image too (only the output is rasterized).
  If users want code+output screenshots, that's a bigger feature with
  its own design.
- Caching based on code content hash — re-running numd always
  regenerates the PNG. Cheap enough for now; revisit if it becomes slow.
- Inline fence-option arguments like `image(theme=xterm, width=80)` —
  configuration goes through the `$env.numd.*` knobs only. Supporting
  inline args would require extending `extract-fence-options` beyond
  comma-splitting, and that complexity is not justified for the first
  pass.
- Special handling for image-producing blocks in `compute-change-stats`
  (line 379). Image references are stable strings, so the existing
  text-level diff is sufficient; no new columns or metrics are needed.

## Post-implementation notes

Tracked separately, not yet actioned — mentioned here so the next
session has the full picture:

- PNGs for `image + try` are byte-non-deterministic across runs
  because the rendered nushell error value embeds span offsets. The
  markdown itself (and therefore `compute-change-stats`) stays
  deterministic, but the committed PNG will show binary diffs on
  every run. Options for later: `.gitignore z_examples/*/media/` or
  strip spans before rasterization.
- Integration harness only diffs `.md` files, not PNGs — a crashed
  `to png` emitting a zero-byte file would not fail the test.
- `$env.numd.image-theme` / `image-font` knobs (in Config knobs) are
  spec'd but not wired. Deferred per the "keep the knob list short
  on the first pass" guidance. The seam is
  `generate-image-output-pipeline` — currently only threads `--width`.
- Pre-existing environmental drift in 5 integration example files
  (`whoami user→agent`, timezone offsets in `working_with_lists.md`)
  is unrelated to this work and was deliberately not committed.
